## 🛠 Опис реалізованого функціоналу

У ході лабораторної роботи було розроблено мобільний додаток з використанням сучасного підходу до навігації — **Expo Router**. Було реалізовано наступні можливості:

- **File-based маршрутизація:** Навігація будується автоматично на основі структури папок і файлів у директорії `app`.
- **Глобальний стан (Context API):** Створено `AuthContext` для централізованого управління станом авторизації користувача (функції входу, реєстрації та виходу).
- **Захищені маршрути (Route Guards):** Реалізовано перевірку стану `isAuthenticated` на рівні `_layout.jsx` групи `(app)`. Неавторизовані користувачі перенаправляються на сторінку логіну за допомогою компонента `<Redirect>`.
- **Логічне розділення (Групи маршрутів):** Використано папки `(auth)` та `(app)` для візуального та архітектурного розділення публічних та приватних екранів, що не впливає на кінцеву URL-адресу.
- **Динамічна навігація:** Створено екран деталей товару `[id].jsx`, який отримує параметри через хук `useLocalSearchParams()`.
- **Обробка помилок (404):** Додано екран `+not-found.jsx` для опрацювання переходів за неіснуючими маршрутами.

## 📚 Висновки (Відповіді на контрольні запитання)

### 1. Яким чином за допомогою Expo Router реалізується перенаправлення неавторизованого користувача?

В Expo Router перенаправлення зручно реалізовувати на рівні Layout-файлів (наприклад, `_layout.jsx` у захищеній групі). За допомогою глобального стану (Context) ми перевіряємо, чи авторизований користувач. Якщо значення `isAuthenticated === false`, ми повертаємо компонент `<Redirect href="/login" />`, який миттєво перекидає користувача на публічний екран, не рендерячи захищений контент.

### 2. У чому полягає різниця між використанням компонента `<Link>` та метода `router.push()`?

- **`<Link>`**: Декларативний спосіб навігації (схожий на тег `<a>` в HTML). Найкраще підходить для статичних посилань в інтерфейсі (наприклад, перехід від списку товарів до деталей). Підтримує атрибут `asChild` для кастомної стилізації.
- **`router.push()` / `router.replace()`**: Імперативний спосіб навігації через об'єкт `router`. Використовується всередині функцій та обробників подій (наприклад, після натискання кнопки "Увійти" та успішної перевірки даних).

### 3. Як працюють динамічні маршрути в Expo Router і як отримати передані параметри?

Динамічні маршрути створюються за допомогою квадратних дужок у назві файлу або папки (наприклад, `app/details/[id].jsx`). Цей синтаксис вказує роутеру, що ця частина шляху є змінною. Щоб отримати значення цієї змінної (наприклад, ID конкретного товару) всередині компонента, використовується спеціальний хук `useLocalSearchParams()`, який повертає об'єкт із параметрами.

### 4. Чому стан авторизації доцільно зберігати у глобальному контексті (React Context), а не в локальному стані компонента?

Стан авторизації є "наскрізним" (global state) і потрібен одночасно в багатьох різних компонентах: у Layout для захисту маршрутів, на головному екрані для відображення імені користувача, в меню для кнопки "Вийти". Якщо зберігати його локально (`useState`), доведеться передавати його через `props` на всі рівні вкладеності (prop drilling), що ускладнює код і робить його нечитабельним. Context вирішує цю проблему, надаючи єдине джерело даних для всього додатка.

### 5. Для чого використовуються групи маршрутів (folderName) і як вони впливають на URL-адресу?

Групи маршрутів (створюються назвою папки в круглих дужках, наприклад `(auth)`) використовуються виключно для організації файлової структури та застосування окремих Layout-файлів до групи екранів. **Вони ігноруються в системі маршрутизації** і не додаються до кінцевого URL. Наприклад, файл `app/(auth)/login.jsx` буде доступний за шляхом `/login`, а не `/(auth)/login`.
